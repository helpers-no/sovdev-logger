# Sovdev Logger Dashboards

Grafana dashboard definitions owned by sovdev-logger itself, and a script to push them to any Grafana instance via its HTTP API.

## Why this exists

The local UIS stack already provisions a dashboard ("Sovdev Logger - Overview", uid `sovdev-metrics`) — but it ships from UIS's own infrastructure repo, not from sovdev-logger, and only covers metrics + basic logs. It has no trace visualization, no job-lifecycle tracking, and no service-dependency view, despite sovdev-logger emitting all of that data today. See [`INVESTIGATE-grafana-dashboard-definitions.md`](https://github.com/helpers-no/sovdev-logger/blob/main/website/docs/ai-developer/plans/completed/INVESTIGATE-grafana-dashboard-definitions.md) for the full audit.

This directory's dashboard (`sovdev-logger-overview.json`, uid `sovdev-logger-full`) is a **separate, sovdev-logger-owned dashboard** — it does not replace or overwrite the UIS-provisioned one. Both can coexist.

## Contents

| File | Purpose |
|------|---------|
| [`sovdev-logger-overview.json`](sovdev-logger-overview.json) | The dashboard itself — hand-authored, not generated from the schema (see [Q3] in the investigation for why) |
| [`push-dashboard.ts`](push-dashboard.ts) | Pushes a dashboard JSON file to a Grafana instance's `/api/dashboards/db` (local UIS; see below for why this isn't used for Grafana Cloud) |
| [`adapt-for-grafana-cloud.ts`](adapt-for-grafana-cloud.ts) | Rewrites the dashboard's datasource UIDs for Grafana Cloud, producing `sovdev-logger-overview-grafana-cloud.json` — see "Deploying to Grafana Cloud" below |
| [`sovdev-logger-overview-grafana-cloud.json`](sovdev-logger-overview-grafana-cloud.json) | Generated by `adapt-for-grafana-cloud.ts` — ready to import as-is via Grafana Cloud's own UI |
| [`configure-trace-correlation.ts`](configure-trace-correlation.ts) | Wires up bidirectional log↔trace correlation (Loki `trace_id` label → Tempo trace; Tempo trace → correlated Loki logs) |

## Usage — local UIS

```bash
npm install   # once, installs tsx/typescript

GRAFANA_URL=http://grafana.localhost GRAFANA_USER=admin GRAFANA_PASSWORD=SecretPassword1 \
  npx tsx push-dashboard.ts
```

Re-running the push is safe — it's the same UID every time (`overwrite: true`), so it updates the existing dashboard rather than creating duplicates.

## Deploying to Grafana Cloud

**`push-dashboard.ts` is deliberately not used for Grafana Cloud.** Pushing via `/api/dashboards/db` needs a Grafana Service Account token (a different credential system than the OTLP ingest/verify Access Policy tokens this project already has) — and per this project's own established rule, minting new credentials is something the maintainer does by hand in the portal, not something scripted or delegated to an agent. Grafana Cloud's own UI already has an equivalent, credential-free path: **Dashboards → New → Import**.

That's not quite a one-command deploy, though — two real gotchas, found by actually doing this once, not assumed:

### Gotcha 1: Import doesn't auto-map datasources for a hand-authored JSON

Grafana's Import wizard only shows an interactive "map each datasource" step when the JSON has an `__inputs`/`__requires` structure — which only gets added when a dashboard is *exported* from Grafana itself via "Export for sharing externally." This dashboard is hand-authored (built by a script, not exported from a UI), so it doesn't have that structure. **Confirmed empirically**: importing it as-is silently keeps the hardcoded local-UIS datasource identifiers (`"prometheus"`, `"loki"`, `"tempo"`), no error at import time — every panel just fails afterward with a small red warning triangle on its title. Hovering that triangle reveals the real error: `Datasource prometheus was not found`. "No data" alone, with no warning triangle, would have meant a genuinely empty query result; the triangle is what distinguishes a real failure from that.

**Fix**: run `adapt-for-grafana-cloud.ts` first, which rewrites every datasource UID to Grafana Cloud's real ones, then import the *generated* file instead of the original:

```bash
npx tsx adapt-for-grafana-cloud.ts
# -> writes sovdev-logger-overview-grafana-cloud.json
```

Then: Grafana Cloud instance → **Dashboards → New → Import → Upload dashboard JSON file** → select `sovdev-logger-overview-grafana-cloud.json`. Since the UIDs are already correct, this import needs no manual mapping step.

### Gotcha 2: the datasource UID is not the display name

`Connections → Data sources` lists datasources by a display name like `grafanacloud-urbalurba-prom` — **that is not the UID** the dashboard JSON needs. The real UID is only visible by clicking into the datasource and reading it off the settings page's URL (`.../connections/datasources/edit/<uid>`). Confirmed on one real stack (`urbalurba`):

| Type | Display name (Data sources list) | Actual UID |
|---|---|---|
| Prometheus | `grafanacloud-urbalurba-prom` | `grafanacloud-prom` |
| Loki | `grafanacloud-urbalurba-logs` | `grafanacloud-logs` |
| Tempo | `grafanacloud-urbalurba-traces` | `grafanacloud-traces` |

These look like Grafana Cloud's fixed, standard UIDs for every stack's auto-provisioned default datasources — plausible given the pattern (no `urbalurba`-specific string in any of them), but only confirmed against this one stack. `adapt-for-grafana-cloud.ts` hardcodes this table; if a different stack's UIDs turn out to differ, update the map at the top of that script and re-run it.

### After importing: verify with fresh data, not old data

This Grafana Cloud stack's Prometheus/Mimir appears to age a short-lived process's metrics out within a few minutes (consistent with the OpenTelemetry Collector's default `metric_expiration` behavior) — checking with stale data produces a false "still broken" signal that looks identical to a real failure (both show "No data", but only one is an actual problem). Push a fresh log/metric through right before checking, the same way [`using/onboarding/index.md`](../../website/docs/using/onboarding/index.md)'s validation step (step 5) does, and confirm via `tools/validation/grafana-cloud/query-prometheus.ts` that it landed before concluding the dashboard itself is broken.

### Log↔trace correlation

```bash
GRAFANA_URL=http://grafana.localhost GRAFANA_USER=admin GRAFANA_PASSWORD=SecretPassword1 \
  npx tsx configure-trace-correlation.ts
```

Also safe to re-run — it removes any correlation it previously created before adding a fresh one, rather than accumulating duplicates (the Grafana Correlations API has no upsert of its own).

sovdev-logger's `trace_id` is a Loki **label**, not text embedded in the log line, so the older `derivedFields` datasource setting (line-text regex only) can't reach it — this uses Grafana's newer Correlations API instead, which links a named field (regardless of whether it's a label or line content) to a Tempo query. The reverse direction uses Tempo's own native `tracesToLogsV2` setting, a different, purpose-built mechanism.

## What's in the dashboard

- **`$service_name` template variable** — every panel is parameterized by it (multi-select, defaults to "All"), replacing what used to be a hardcoded service-name regex in the UIS-provisioned dashboard
- **Metrics** (ported from the existing dashboard): Active Integrations, Total Operations, Error Rate, Average Operation Duration
- ~~**Operations / Errors / Error Rate / Avg Duration by Peer Service**~~ — **removed** (2026-07-13). These 4 Stat panels turned out to be redundant with the "Metrics" panels above: `Total Operations`/`Error Rate`/`Average Operation Duration` already break down `by (service_name, peer_service)` and render every combination via their table-mode legend (`{{service_name}} → {{peer_service}}`), so a separate "by Peer Service" re-slice added no information — just the same data with `service_name` dropped and rendered as Stat tiles instead. They also carried the `maxDataPoints: 1` fragility documented in the parent plan's "Post-completion fix" section (a single sample point can miss a short-lived test process's metrics if they've aged out of the window by the time it's taken), which the multi-point `timeseries` panels above don't have. Removed rather than re-fixed a second time.
- **Job Lifecycle** (new) — a logs panel showing `log_type="job.status"`/`"job.progress"` entries in chronological order, so a batch job's start → progress → completion reads as a natural timeline
- **Recent Errors** / **Transaction Logs** (ported from the existing dashboard) — detailed Loki tables/logs for `log_type="transaction"` entries. **Recent Errors** includes a "Client" column showing `client_name` (set via `sovdev_set_context()`, see `PLAN-context-propagation.md`) — like `peer_service`, this is not a real Loki label (it's per-request structured metadata, not a per-deployment resource attribute), so it can't be a `$service_name`-style template variable; it's surfaced the same way `peer_service` already was, via the table's `extractFields` transform, not a filterable dropdown.

Traces (Tempo) and log↔trace correlation are Phase 2 of the parent plan, not yet in this dashboard.
